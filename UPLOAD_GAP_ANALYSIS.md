# DCBrain — Upload Gap Analysis

DCBrain exposes exactly **three** user-facing upload/import controls. Every other module derives its data from these three or from in-app generation. This is by design, but it means several disciplines have **no dedicated upload path** and rely on the Documents RAG store.

## Modules WITH upload support

| Module | Route | Control | Accepts |
|--------|-------|---------|---------|
| Documents | `/documents` | Drag-drop modal, `files[]` (≤50/batch) | pdf, docx, xlsx, csv, json, xml, txt, md, png, jpg, jpeg, tif, tiff |
| Procurement | `/procurement` | Single-file import, `file` | csv, xlsx (rows keyed on Material + Vendor) |
| Schedule Risk | `/schedule` | Single-file / drag-drop import, `file` | xml (Primavera P6) |

## Modules WITHOUT upload support (data is derived)

| Module | Route | How it receives data | Depends on |
|--------|-------|----------------------|-----------|
| Compliance | `/compliance` | Runs standards checks over uploaded documents | Documents |
| RFIs | `/rfis` | Created in-app; AI suggests answers from RAG | Documents |
| NCRs | `/ncrs` | Created in-app | — |
| Inspections | `/inspections` | Created in-app | — |
| Commissioning | `/commissioning` | Created in-app / derived | Documents |
| Quality | `/quality` | Aggregated from QA/QC records | Documents |
| Change Orders | `/change-orders` | Created in-app | — |
| Reports | `/reports` | Generated by the platform (PDF/MD) | all modules |
| Simulations | `/simulations` | Run over imported schedule | Schedule Risk |
| Graph | `/graph` | Neo4j dependency/failure graph | Schedule + Procurement |
| Agents | `/agents` | Triggered by uploads/imports | all modules |

## Identified gaps & recommendations

### 1. Procurement importer schema mismatch (HIGH)
The procurement importer keys each row on a `Material` / `Vendor` column. Several demo tables use different headers (`description`, `equipment_id`, banner title rows in XLSX). Those files are therefore routed to **Documents** (so they are still indexed for RAG) rather than the procurement pipeline, and only the schema-matching tables are imported.

- **Affected:** procurement registers, summaries, matrices, and PDFs (65 procurement files routed to Documents).
- **Recommendation:** add a header-mapping / preview step to `/procurement` import, or normalise the demo CSVs to the importer schema (`Material`, `Vendor`, `PO Number`, `Quantity`, `Status`, `Required On Site Date`).
- **Effort:** ~0.5 day (backend header-alias map) or ~1 day (UI column mapper).

### 2. No bulk/folder upload for Documents (MEDIUM)
Documents upload is capped at 50 files per batch and has no recursive/folder mode. Large disciplines must be uploaded in multiple batches (the automation handles this transparently).
- **Recommendation:** optional server-side zip ingestion, or a "upload folder" affordance.
- **Effort:** ~1–2 days.

### 3. No direct upload for Compliance / RFI evidence (LOW)
Compliance and RFI modules cannot attach source files directly; they reference documents already in the RAG store. This is acceptable given the architecture — uploading the compliance/RFI PDFs to Documents makes them available to those modules.
- **Recommendation:** none required for the demo; consider per-module attachment in future.

> **Conclusion:** no new features are required to upload the full dataset. The automation routes every file to the correct existing control, imports schema-matching procurement tables, and indexes the remaining procurement tables + all discipline documents through Documents.
